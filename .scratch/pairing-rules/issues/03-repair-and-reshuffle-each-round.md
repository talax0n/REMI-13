# Re-pair and reshuffle every round of penyisihan

## Original feedback

> Setiap babak harus pairing ulang dan formasi pemain harus di acak ulang

## Summary

Each penyisihan round must regenerate tables from scratch — same player should not face the same opponents twice if avoidable. Verify current `shuffle-engine` does this or carries over previous tables.

## Agent Brief

**Category:** bug
**Summary:** Ensure every penyisihan round re-pairs active players, persists opponent history, and has regression coverage for avoiding repeat opponents when possible.

**Current behavior:**
The shuffle engine already supports randomized table generation and repeat-opponent minimization using each participant's prior opponent history. The admin shuffle flow for regular penyisihan rounds and semi-final rounds calls that engine and records newly generated opponents on the successful path.

I could not reproduce a simple “tables are carried over unchanged” failure from the current normal code path. Current table display rows are replaced, not reused.

There is a reproducible data-flow bug around the seeded first round: first boot/reset creates phase-1 tables from existing table numbers but initializes participants with empty opponent history. The first generated reshuffle therefore has no knowledge of phase-1 tablemates, so repeat meetings from phase 1 into phase 2 are possible and not counted.

The behavior is also not protected by tests, and there is another risky path: if regular penyisihan shuffling falls back to a plain randomized grouping after an engine error, the new table assignments are produced without recording opponent history for that round. That means later rounds may not know those players already met.

**Desired behavior:**
Every penyisihan round after the initial assignment should generate fresh table assignments from the current active player pool. The implementation should not reuse the prior round's table grouping unless the constraints make an identical grouping unavoidable.

After each generated round, all tablemates should be added to each participant's opponent history exactly once for that round, regardless of whether the engine path or any fallback path produced the tables. Later rounds should use that history to minimize repeat tablemates when an alternative valid arrangement exists.

The seeded phase-1 assignment should also be recorded as prior opponent history before the first generated reshuffle uses it. This can happen during initialization, before the first shuffle, or through another durable mechanism, as long as the first generated reshuffle can avoid phase-1 tablemates where possible.

Table numbers may reset each round. The important invariant is player-table membership, not continuity of table numbers.

**Key interfaces:**
- Shuffle engine participant input — must receive the current active participants with their accumulated opponent history.
- Shuffle result — should expose generated table groups and repeat-opponent diagnostics.
- Opponent-history update function — should be called once after each generated round, including fallback generation paths.
- Admin shuffle action — should persist updated table assignments and updated opponent history together.
- Participant persistence shape — should preserve each participant's opponent IDs across phase transitions and reloads.

**Acceptance criteria:**
- [ ] A penyisihan shuffle for a later round generates table groups from the current active player pool instead of reusing the previous table grouping.
- [ ] Seeded phase-1 tablemates are recorded into opponent history before the first generated reshuffle.
- [ ] After a round is generated, every participant has the other players from that round's table added to their opponent history.
- [ ] Opponent history is updated on all table-generation paths, including fallback grouping if the primary engine throws.
- [ ] A regression test or deterministic verification covers phase 1 into phase 2 and proves seeded phase-1 opponent history is considered by the first generated reshuffle.
- [ ] A regression test or deterministic verification covers two consecutive generated penyisihan shuffles and proves opponent history from the first generated round is considered by the second.
- [ ] When a no-repeat arrangement is possible, the second generated round avoids repeat tablemates.
- [ ] When repeat tablemates are mathematically unavoidable, the result remains valid and reports/handles the repeat count instead of silently carrying over tables.
- [ ] Table numbering can restart at 1 each generated round.
- [ ] Running lint/type checks succeeds.

**Out of scope:**
- Adding dummy players for non-multiples of 5; that belongs to the fixed-table-size issue.
- Implementing the same-team-different-tables rule; that belongs to the same-team pairing issue.
- Changing semi-final or final cutoff policy.
