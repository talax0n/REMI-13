## Agent Brief

**Category:** enhancement
**Summary:** Rename the participant grouping concept from Church to Team across storage, APIs, types, CSV import, pairing logic, and user-facing UI.

**Current behavior:**
Participants are grouped by a `church` field throughout the app. The database stores `players.church`, app types expose `church`, admin and player APIs accept and return `church`, CSV import requires a `church` column, and UI labels say “Church”. Pairing logic also describes same-group constraints with Church/Gereja terminology. This conflicts with current event language, where the grouping should be called “Team”.

**Desired behavior:**
`team` should be the canonical field and vocabulary for participant grouping. New app code, public API responses, admin/player payloads, CSV import instructions, UI labels, validation messages, search/filter labels, styling helpers, and pairing warnings should use “Team”/`team` consistently.

Existing data should be preserved. If an existing database has a `church` column, migration should carry those values into the canonical `team` field rather than dropping them. Stored table snapshots that contain participant objects should also be normalized so future reads expose `team`.

For a temporary compatibility window, legacy inputs using `church` should still be accepted where users or older clients may provide them:

- Player lookup may accept either `team` or legacy `church`, while responses should expose `team`.
- Admin participant sync/import may accept legacy `church`, while persisted and returned participant objects should expose `team`.
- CSV import should prefer a `team` column but may accept `church` as a legacy alias with copy that tells admins the required column is now `team`.

Visible labels should use “Team”. Do not keep “Church” as the primary user-facing label. Existing actual team names such as GPIB names are data values and should not be rewritten.

**Key interfaces:**
- `AdminParticipant` grouping field — should expose `team` instead of `church`.
- Public display `Player` shape — should expose `team` instead of `church`.
- `PlayerScore` and `LoginFormData` — should use `team` for lookup and display.
- Player lookup API contract — should prefer `team`, with temporary `church` alias support.
- Admin participant sync contract — should prefer `team`, with temporary `church` alias support.
- Database player record shape — should preserve existing grouping values while making `team` canonical.
- Stored table participant JSON shape — should read/write `team` consistently.
- CSV import schema — should prefer `name,team`; legacy `name,church` should still parse during the compatibility window.
- Pairing constraint result/warning names — should use Team terminology so this issue aligns with same-team pairing constraints.

**Acceptance criteria:**
- [ ] New TypeScript participant/player/login types use `team` as the canonical grouping field.
- [ ] Database migration preserves any existing `church` values in the canonical `team` field.
- [ ] Player and admin APIs return participant objects with `team`, not `church`.
- [ ] Player and admin APIs accept legacy `church` input as an alias during the compatibility window.
- [ ] CSV import copy and validation prefer a `team` column.
- [ ] CSV import still accepts legacy `church` column data and maps it to `team`.
- [ ] Admin participant form, participant table filters/sort headers, bulk import preview, player login, leaderboard, table display, and scoring views show “Team” instead of “Church”.
- [ ] Pairing code and warnings use Team terminology for same-group constraints.
- [ ] Existing team names in seed/sample data remain unchanged as data values.
- [ ] Running lint/type checks succeeds after the rename.

**Out of scope:**
- Implementing the same-team-different-tables pairing rule itself; that belongs to the pairing-rules issue.
- Renaming actual organization/team names stored as participant data.
- Removing the temporary legacy `church` input alias before downstream callers have moved to `team`.
